# 后端问题排查

后端服务是系统的核心，负责数据处理、设备连接和业务逻辑。在排查后端问题前，请确保环境层和网络层已验证通过。

## 诊断前自查清单
- 环境问题是否已排查？Python版本、依赖版本是否正确？
- 后端服务启动时是否有错误日志？完整的错误堆栈是什么？
- 后端配置文件（如.env、config.yaml）是否正确配置？
- 数据库、Home Assistant、MQTT等外部服务是否已启动？
- 最近是否修改过后端代码或配置？

---

## 一、启动失败问题

### 现象特征
- 运行启动命令后立即退出
- 终端输出错误信息
- 进程启动后很快崩溃
- 端口未被监听

### 引导式排查

1. **启动命令与入口**
   - 你使用的启动命令是什么？是否与文档一致？
   - 入口文件是否存在？路径是否正确？
   - 工作目录是否正确？（Python模块导入依赖工作目录）
   - 尝试直接用Python运行入口文件，看是否有更详细的错误输出

2. **配置加载问题**
   - 程序是否能找到配置文件？配置文件路径配置是否正确？
   - 配置文件格式是否正确？（YAML缩进、JSON语法、环境变量格式）
   - 必填配置项是否都已填写？是否有缺失的配置导致校验失败？
   - 尝试打印加载的配置，确认配置值是否符合预期
   - 配置文件中是否有语法错误？（如多余的逗号、错误的缩进）

3. **导入与初始化错误**
   - 错误堆栈中是否有`ImportError`或`ModuleNotFoundError`？
   - 是哪个模块导入失败？该模块是否已安装？
   - 是否存在循环导入问题？
   - 初始化代码中是否有变量未定义、函数调用参数错误？
   - 是否在模块级别执行了数据库连接等可能失败的操作？

4. **端口绑定问题**
   - 错误信息中是否有"Address already in use"或类似提示？
   - 配置的端口号是多少？该端口是否被其他进程占用？
   - 是否尝试绑定到一个需要root权限的端口（<1024）？
   - 绑定的IP地址是否正确？（0.0.0.0 vs 127.0.0.1）

### 验证方法
- 开启debug模式运行，获取更详细的日志
- 注释掉部分初始化代码，逐步缩小问题范围
- 检查日志文件是否有更完整的错误记录

## 二、数据库连接问题

### 现象特征
- 启动时提示数据库连接失败
- 数据库操作超时
- 数据无法保存或查询
- 日志中出现SQLite/MySQL/PostgreSQL相关错误

### 引导式排查

1. **数据库配置验证**
   - 使用的是什么数据库？（SQLite/MySQL/PostgreSQL等）
   - 数据库连接字符串是否正确？
   - 数据库地址、端口、用户名、密码是否填写正确？
   - 数据库名称是否存在？

2. **SQLite特有问题**
   - 数据库文件路径是否正确？程序是否有写权限？
   - 数据库文件所在目录是否存在？
   - 是否有其他进程同时锁定了数据库文件？
   - 数据库文件是否损坏？

3. **MySQL/PostgreSQL等服务端数据库**
   - 数据库服务是否已启动？能否通过命令行客户端连接？
   - 从后端所在机器能否telnet/nc到数据库端口？
   - 数据库用户是否有足够的权限？
   - 防火墙是否允许后端服务器访问数据库端口？
   - 数据库是否配置了允许远程连接？（如果不在同一台机器）
   - 是否达到了数据库最大连接数限制？

4. **连接池与超时**
   - 连接池配置是否合理？（最大连接数、超时时间）
   - 是否存在连接泄漏？（连接未正确释放）
   - 网络延迟是否过高导致连接超时？
   - 数据库负载是否过高响应缓慢？

### 验证方法
- 使用数据库客户端工具，用相同的连接参数手动连接测试
- 查看数据库日志，确认是否有连接记录或错误
- 简化数据库操作，测试最简单的连接和查询是否正常

## 三、Home Assistant连接问题

### 现象特征
- 无法发现或连接Home Assistant
- 实体状态无法同步
- 调用HA服务失败
- WebSocket连接断开

### 引导式排查

1. **HA访问配置**
   - Home Assistant的访问地址是否正确？（IP/域名、端口）
   - 使用的是HTTP还是HTTPS？证书是否有效？
   - Long-Lived Access Token是否正确生成？
   - Token是否有过期或被撤销？
   - 从后端服务器能否正常访问HA的Web界面？

2. **API连接测试**
   - 尝试直接curl HA的API接口，验证是否能正常访问：
     ```bash
     curl -X GET -H "Authorization: Bearer YOUR_TOKEN" http://HA_IP:8123/api/
     ```
   - 返回结果是什么？是401未授权、404找不到还是连接超时？
   - HA的`configuration.yaml`中是否配置了CORS或其他访问限制？

3. **WebSocket连接问题**
   - WebSocket连接URL是否正确？（ws:// vs wss://）
   - 连接建立后是否有认证失败的消息？
   - 是否有频繁断开重连的情况？断开原因是什么？
   - HA端是否有相关错误日志？

4. **实体与服务问题**
   - 要操作的实体ID是否正确存在？在HA开发者工具中确认
   - 调用的服务名称是否正确？参数格式是否符合要求？
   - 相关的HA集成是否已正确配置并运行？
   - HA本身是否正常运行？有没有重启或更新？

### 验证方法
- 查看后端日志中HA相关的错误信息
- 在HA的日志页面查找是否有来自API的错误请求
- 使用HA的开发者工具→服务功能手动测试服务调用

## 四、MQTT连接问题

### 现象特征
- MQTT客户端连接失败
- 无法订阅主题或发布消息
- 消息收发延迟或丢失
- 频繁断开重连

### 引导式排查

1. **MQTT Broker配置**
   - MQTT Broker地址和端口是否正确？（默认1883或8883 for TLS）
   - Broker服务是否已启动？
   - 用户名密码是否正确（如果Broker开启了认证）？
   - 使用的是MQTT over TCP还是WebSocket？端口是否对应？

2. **网络连通性**
   - 从后端服务器能否telnet/nc到Broker的IP和端口？
   - 防火墙是否放行了MQTT端口？
   - 是否有网络ACL或安全组限制？
   - Broker是否绑定到了正确的网络接口？

3. **TLS/SSL问题（如果使用加密连接）**
   - CA证书是否正确配置？
   - 证书是否过期？
   - Broker地址与证书中的域名是否匹配？
   - 是否跳过了证书验证但配置要求验证？

4. **主题与消息问题**
   - 订阅的主题格式是否正确？是否有拼写错误？
   - QoS级别设置是否合理？
   - 发布消息的客户端是否有权限发布到该主题？
   - 是否有其他客户端同时订阅/发布导致消息被消费？
   - MQTT Broker的ACL配置是否限制了主题访问？

5. **客户端配置问题**
   - Client ID是否重复？（重复ID会导致互相踢下线）
   - Keepalive间隔设置是否合理？
   - 遗嘱消息（LWT）配置是否正确？
   - 自动重连是否配置？重连间隔是否合理？

### 验证方法
- 使用命令行MQTT客户端（如mosquitto_sub/mosquitto_pub）测试连接和消息收发
- 查看MQTT Broker的日志，确认连接情况
- 使用Wireshark抓包分析MQTT连接过程
- 简化QoS为0，不使用TLS，先确保基本连接正常

## 五、运行时异常问题

### 现象特征
- 服务启动正常但运行过程中报错
- 某些API接口调用失败
- 内存占用持续增长
- CPU占用过高
- 请求响应缓慢

### 引导式排查

1. **错误日志分析**
   - 错误发生在哪个模块/接口？
   - 错误堆栈的完整信息是什么？第一个报错的位置在哪里？
   - 是否有规律？比如特定操作后必现？
   - 是偶发还是必现？

2. **API接口问题**
   - 请求的URL、方法、参数是否正确？
   - 请求参数是否符合接口定义？类型、范围是否正确？
   - 是否缺少必要的请求头？（如Content-Type、Authorization）
   - 请求体格式是否正确？（JSON、表单等）
   - 接口返回的状态码和错误信息是什么？

3. **性能问题**
   - CPU/内存/磁盘IO/网络IO哪项资源占用高？
   - 是否有死循环或大量计算？
   - 是否有数据库慢查询？
   - 是否有外部服务调用超时未设置超时时间？
   - 并发量是否过高？是否有资源竞争？
   - 是否有内存泄漏？（对象未释放、缓存无限增长）

4. **日志级别调整**
   - 将日志级别调整为DEBUG，获取更详细的运行信息
   - 在关键位置添加日志，追踪执行流程
   - 注意不要在生产环境一直开启DEBUG级别，会影响性能

### 排查思路建议
- 问题复现是关键：尝试找到稳定复现问题的步骤
- 二分法定位：注释/回退部分代码，确定是哪部分代码引入的问题
- 对比验证：对比正常运行和异常时的状态差异
- 不要忽略警告：有时候WARNING级别的日志已经预示了问题
