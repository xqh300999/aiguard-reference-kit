# 网络问题排查

网络是分布式系统的通信基础。在排查网络问题时，建议从底层到上层逐步验证：物理连接→IP层→传输层（TCP/UDP）→应用层协议。

## 诊断前自查清单
- 网线/WiFi是否正常连接？
- 同一局域网内的设备能否互相ping通？
- 问题是所有网络都不通还是某个特定端口/服务不通？
- 最近是否修改过防火墙、路由器、网络配置？
- 是一开始就不通还是之前正常现在突然不通？

---

## 一、端口占用问题

### 现象特征
- 服务启动时提示"Address already in use"、"端口被占用"
- 服务启动后无法访问对应端口
- 服务启动了但响应的是另一个应用
- 关闭服务后端口仍被占用（TIME_WAIT状态）

### 引导式排查

1. **端口占用确认**
   - 服务配置的端口号是多少？
   - 你确定是这个端口被占用吗？错误信息中明确指出端口号了吗？
   - 是TCP端口还是UDP端口？某些服务同时使用TCP和UDP
   - 是IPv4端口还是IPv6端口？

2. **查找占用端口的进程**
   - 使用系统命令查看端口占用情况：
     - Windows: `netstat -ano | findstr :端口号`，然后用`tasklist | findstr PID`查找进程
     - macOS/Linux: `lsof -i :端口号` 或 `netstat -tulpn | grep :端口号`
   - 占用端口的进程是什么？是你预期的进程吗？
   - 是之前启动的服务没正常退出吗？
   - 是另一个无关的程序占用了这个端口吗？

3. **端口占用类型**
   - 是同端口上已有服务在监听？
   - 还是端口处于TIME_WAIT状态？（刚关闭的连接会等待一段时间）
   - 如果是TIME_WAIT，可以等待一段时间或配置端口复用
   - 是否有多个服务实例同时启动了？

4. **解决思路验证**
   - 能否关闭占用端口的其他程序？
   - 能否将你的服务配置为使用其他端口？
   - 检查配置文件，是否重复配置了同一个端口？
   - 对于开发环境，是否开启了自动重载导致多个进程并存？

### 验证方法
- 找到并结束占用端口的进程，重新启动你的服务
- 更换服务端口，看是否能正常启动和访问
- 使用`telnet IP 端口`或`nc -zv IP 端口`测试端口是否开放

## 二、防火墙问题

### 现象特征
- 本机可以访问服务，其他机器访问不了
- 能ping通但端口连不上
- 某些端口可以访问，某些不行
- 关闭防火墙后正常，开启后异常

### 引导式排查

1. **防火墙状态确认**
   - 操作系统自带防火墙是否开启？（Windows Defender防火墙、macOS防火墙、iptables/ufw/firewalld）
   - 是否安装了第三方安全软件/杀毒软件自带防火墙？
   - 云服务器的安全组/网络ACL是否放行了相应端口？
   - 路由器/网关是否有端口访问限制？

2. **防火墙规则检查**
   - 入站规则（Inbound）是否允许服务端口的访问？
   - 出站规则（Outbound）是否限制了对外访问？
   - 规则是针对哪个网络位置的？（域/专用/公用网络）
   - 规则允许的源IP范围是什么？是否限制了只能本地访问？
   - 规则的协议是TCP还是UDP？是否和服务使用的协议一致？
   - 规则是否启用？优先级是否正确？

3. **系统级验证**
   - 在服务所在机器本地测试访问，看是否正常？
   - 在另一台机器上测试访问，失败吗？
   - 临时关闭防火墙测试一下，如果关闭后正常，说明确实是防火墙问题
   - 查看防火墙日志，看是否有被拦截的记录

4. **端口监听验证**
   - 服务是监听在127.0.0.1还是0.0.0.0？
   - 如果只监听127.0.0.1，只能本机访问，其他机器无法连接
   - 确认服务绑定地址配置正确，需要外部访问应绑定0.0.0.0
   - 用`netstat`/`ss`命令查看监听地址：`netstat -tulpn | grep 端口号`

### 验证方法
- 本地使用`curl http://localhost:端口`测试能否访问
- 从另一台机器使用`curl http://服务IP:端口`测试
- 临时关闭防火墙验证连通性（注意安全，测试完及时开启并正确配置规则）
- 用nmap从外部扫描端口，看端口是open/filtered/closed状态

## 三、设备发现问题

### 现象特征
- 设备连接到WiFi但应用发现不了
- mDNS设备无法通过.local域名访问
- SSDP/UPnP设备搜索不到
- 同一局域网内但看不到对方

### 引导式排查

1. **局域网连通性基础**
   - 设备和手机/电脑是否真的在同一个局域网？
   - 检查两者的IP地址，是否在同一个网段？（如都是192.168.1.x）
   - 路由器是否开启了AP隔离/客户端隔离？这会禁止设备之间互相通信
   - 能否互相ping通对方的IP地址？
   - 是否连接了同一个SSID？（注意2.4G和5G可能是不同的网段）

2. **mDNS/Bonjour发现问题**
   - 设备是否正常启动并注册了mDNS服务？
   - 电脑/手机是否支持mDNS？
     - Windows需要安装Bonjour Print Services
     - macOS自带支持
     - Linux需要安装avahi-daemon
   - 能否通过`设备名.local`直接ping通设备？
   - 使用mDNS浏览器工具（如Bonjour Browser、Discovery）查看设备是否广播服务
   - 路由器是否阻止了组播数据包（mDNS使用224.0.0.251:5353组播）？

3. **广播/组播问题**
   - 设备发现通常使用广播或组播
   - 交换机/路由器是否开启了广播风暴抑制，把发现包拦截了？
   - 网络是否支持UDP广播？
   - IoT设备和发现端是否在同一个VLAN/子网？广播通常不能跨子网

4. **手动连接测试**
   - 如果自动发现失败，尝试手动输入设备IP地址连接
   - 如果手动输入IP可以连接，说明问题出在发现机制
   - 如果手动输入也连接不上，说明是基础网络连通性问题
   - 检查设备上的服务是否正常运行在预期端口

### 验证方法
- 使用IP扫描工具（如Advanced IP Scanner、nmap、fing）扫描局域网，确认设备在线
- 使用Wireshark抓包，看设备是否发出了广播/组播包，以及应用是否收到了响应
- 检查路由器设置，确认没有AP隔离、VLAN隔离

## 四、WebSocket连接问题

### 现象特征
- WebSocket连接失败
- 连接建立后很快断开
- 消息收发延迟或丢失
- 频繁自动重连
- 前端报WebSocket错误

### 引导式排查

1. **连接建立问题**
   - WebSocket URL是否正确？是ws://还是wss://？
   - 端口是否正确？是否和后端监听端口一致？
   - 后端是否正确配置了WebSocket支持？
   - HTTP握手阶段返回的状态码是多少？应该是101 Switching Protocols
   - 如果返回404，检查路径是否正确；如果返回403，检查是否有跨域或认证问题

2. **反向代理问题（Nginx/Apache）**
   - 如果有反向代理，是否正确配置了WebSocket升级？
   - Nginx需要配置：
     ```
     proxy_set_header Upgrade $http_upgrade;
     proxy_set_header Connection "upgrade";
     ```
   - 代理超时时间配置是否过短？导致长连接被切断
   - 是否配置了正确的proxy_read_timeout？
   - 是否有负载均衡导致连接被分配到不同后端？需要会话保持吗？

3. **连接断开问题**
   - 断开时的错误码和原因是什么？
     - 1000：正常关闭
     - 1006：异常断开（网络问题、服务重启）
     - 1011：服务器错误
     - 1012/1013：服务重启/过载
   - 是网络空闲超时断开吗？是否配置了心跳/ping-pong？
   - 后端服务是否重启或崩溃导致连接断开？
   - 网络是否不稳定，导致TCP连接中断？
   - 运营商/防火墙是否会清理空闲连接？

4. **消息收发问题**
   - 消息是发送了但对方没收到，还是根本发不出去？
   - 消息格式是否正确？JSON是否合法？
   - 消息大小是否超过限制？
   - 后端是否正确订阅/监听了消息事件？
   - 是否有跨域导致浏览器不允许发送？
   - 连接状态是否是OPEN时才发送消息？

5. **认证与安全问题**
   - WebSocket连接是否需要认证？Token如何传递？
   - 是在URL参数中带Token还是在协议头中？
   - wss://连接证书是否有效？自签名证书可能被浏览器拒绝
   - 检查Origin头，后端是否验证Origin防止跨站WebSocket劫持？

### 验证方法
- 使用wscat/websocat等命令行工具测试WebSocket连接，排除浏览器问题
- 浏览器开发者工具Network标签→WS，查看WebSocket握手和消息帧
- 抓包查看WebSocket握手过程和断开原因
- 后端开启WebSocket调试日志，查看连接和消息日志

## 五、通用网络排查工具与命令

### 基础连通性测试
- `ping IP/域名`：测试IP层连通性和延迟
  - 如果ping不通，问题在IP层或更低层
  - 如果ping得通但服务访问不了，问题在传输层或应用层
  - 注意：某些设备会禁用ICMP，ping不通不代表一定不在线

- `telnet IP 端口` / `nc -zv IP 端口`：测试TCP端口连通性
  - 连接成功表示端口开放，网络可达
  - 连接超时/拒绝表示端口不通或被防火墙拦截

### 路由与路径排查
- `traceroute IP`（macOS/Linux）/ `tracert IP`（Windows）：查看数据包经过的路由路径
- `ipconfig`（Windows）/ `ifconfig`/`ip addr`（macOS/Linux）：查看本机IP配置
- `arp -a`：查看ARP表，确认同网段设备MAC地址是否正确解析

### DNS测试
- `nslookup 域名` / `dig 域名`：测试DNS解析是否正常
- 直接用IP访问测试，如果IP可以但域名不行，是DNS问题
- 检查hosts文件是否有错误配置

### 抓包分析
- **Wireshark**：图形化抓包工具，功能强大，可以分析各层协议
- **tcpdump**：命令行抓包工具，适合在服务器上使用
- 抓包可以看到：TCP三次握手是否成功、连接是否被RST、数据是否发送、是否有重传
- 抓包技巧：先抓ICMP（ping）确认基础连通，再抓特定端口的TCP/UDP流量

### 网络问题排查思路
1. **分层排查**：物理层→数据链路层→网络层（IP）→传输层（TCP/UDP）→应用层
2. **两端对比**：在客户端和服务器两端同时抓包对比，看数据包在哪里丢失了
3. **逐步验证**：从本机到同设备、到同网段、到跨网段、到外网，逐步扩大范围
4. **控制变量**：一次只改变一个条件，确定哪个因素影响了结果
5. **替代验证**：用不同客户端、不同网络、不同服务器测试，排除单点问题
