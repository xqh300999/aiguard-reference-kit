# 故障排查指南

## 排查方法论

故障排查是一个系统性的问题诊断过程。本指南采用分层排查方法论，引导你从底层到上层逐步定位问题根源，培养独立诊断和解决问题的能力。

### 分层排查原则

我们按照以下层级顺序进行排查，每一层验证通过后再进入下一层：

```
环境层 → 网络层 → 后端层 → 前端层 → 硬件层
```

**为什么要按这个顺序？**
- 环境问题是最常见也是最容易被忽略的问题
- 网络是所有组件通信的基础
- 后端服务依赖正确的环境和网络配置
- 前端依赖后端服务正常运行
- 硬件问题通常需要在软件层面验证后再排查

### 引导式排查流程

每个问题都遵循以下诊断步骤：

1. **现象观察**：仔细记录问题的具体表现，包括错误信息、日志输出、异常行为
2. **假设验证**：根据现象提出可能的原因假设，设计验证方法
3. **信息收集**：收集相关日志、配置、版本信息
4. **逐步排查**：按照分层方法论，从最可能的层级开始验证
5. **问题定位**：通过排除法确定根本原因
6. **解决验证**：实施解决方案并验证问题是否解决

### 使用本指南

遇到问题时：
1. 先阅读本文档了解排查思路
2. 根据问题表现判断属于哪个层级
3. 查阅对应层级的排查文档
4. 按照文档中的引导问题逐步自查
5. 如果在某一层级未发现问题，继续向上一层排查
6. 如果多个层级都没有问题，查阅FAQ常见问题

### 诊断信息收集清单

在开始排查前，请准备好以下信息：
- 操作系统版本和架构
- Python/Node.js等运行时版本
- 相关依赖包版本
- 完整的错误日志和堆栈信息
- 问题复现步骤
- 最近的配置或代码变更记录

---

**注意**：本指南不会直接给出唯一答案，而是通过提问和引导帮助你自主诊断。每个问题的原因可能因环境而异，掌握排查方法比记住解决方案更重要。
