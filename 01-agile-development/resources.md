## 书籍推荐

### 入门必读

- [推荐] **《Scrum 敏捷开发》**（Scrum and XP from the Trenches）- Henrik Kniberg
  - 实战经验总结，非常接地气，薄薄一本但信息量很大
  - 适合快速理解 Scrum 在真实项目中怎么用

- [推荐] **《敏捷软件开发：原则、模式与实践》** - Robert C. Martin
  - 敏捷原则的经典著作，不仅讲 Scrum，还讲敏捷价值观和技术实践

### 深入学习

- [推荐] **《用户故事与敏捷方法》**（User Stories Applied）- Mike Cohn
  - 详细讲解如何写好用户故事，如何做估算和规划

- [推荐] **《敏捷估计与规划》**（Agile Estimating and Planning）- Mike Cohn
  - 敏捷估算和计划的实践指南

- [推荐] **《Sprint 制胜：如何做一个卓越的 Scrum Master》**（Coaching Agile Teams）- Lyssa Adkins
  - 适合 Scrum Master 和教练角色

### 看板与精益

- [推荐] **《看板方法》**（Kanban）- David J. Anderson
  - 如果想了解看板方法，可以与 Scrum 结合使用

- [推荐] **《精益创业》**（The Lean Startup）- Eric Ries
  - 精益思想，构建-度量-学习循环

## 常用敏捷术语表

| 术语 | 英文 | 说明 |
|------|------|------|
| 产品待办列表 | Product Backlog | 产品所有需求的有序列表 |
| Sprint 待办列表 | Sprint Backlog | 当前 Sprint 要完成的任务列表 |
| 用户故事 | User Story | 从用户视角描述需求的方式，通常格式：作为...我想要...以便... |
| 故事点 | Story Point | 相对工作量估算单位，不是时间 |
| 产品负责人 | Product Owner (PO) | 负责产品价值最大化、Backlog 排序的人 |
| Scrum 主管 | Scrum Master (SM) | 负责确保 Scrum 被正确实施、移除障碍的人 |
| 每日站会 | Daily Standup / Daily Scrum | 每天 15 分钟的团队同步会议 |
| Sprint 评审 | Sprint Review | Sprint 结束时演示成果、收集反馈的会议 |
| Sprint 回顾 | Sprint Retrospective | Sprint 结束后反思过程、改进工作方式的会议 |
| 增量 | Increment | Sprint 交付的可工作的产品 |
| 完成的定义 | Definition of Done (DoD) | 团队约定的「完成」标准 |
| 燃尽图 | Burndown Chart | 展示 Sprint 剩余工作量的图表 |
| 速度 | Velocity | 团队每个 Sprint 完成的故事点数，用于预测 |

## 学习建议

1. **从 Scrum Guide 开始**：先读官方 Scrum Guide（十几页），建立基础概念
2. **边做边学**：敏捷是实践出来的，不是看书看会的。在项目中用，在回顾中改进
3. **不要照搬**：没有完美的流程，适合你们团队的才是最好的。模板是参考，不是教条
4. **持续改进**：每次回顾会改进一点点，几个 Sprint 后效果就会很明显
5. **鼓励创新**：如果你有更好的实践方式，大胆尝试！敏捷本身就是拥抱变化的。
