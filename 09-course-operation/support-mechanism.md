# 协作支持机制

> **用途**：本文档介绍 AiGuard 小学期项目中的团队互助、共性问题沉淀和答疑节奏建议。它不是固定流程，可根据团队实际情况调整。

## 支持思路

项目问题可以先按以下路径流转：

```text
个人自查 → 组内排查 → 跨组互助 → 集中答疑 → FAQ 沉淀
```

这样做的目的，是让问题定位过程也成为学习成果，而不仅仅是得到一个答案。

## 问题自查

提交 Issue 前，建议先准备以下信息：

- 想完成什么目标？
- 当前现象是什么？
- 是否能稳定复现？
- 关键日志或截图在哪里？
- 已经尝试过哪些排查？
- 可能属于哪一层：环境、网络、后端、前端、硬件？
- 是否阻塞 SOS 主链路或影响多人联调？

## 共性问题

如果同类问题出现在多个小组，适合整理为 FAQ 或短讲材料。例如：

- MQTT Topic 与 Payload 不一致
- 后端订阅成功但没有收到消息
- 数据库连接失败
- 前端接口跨域或字段不一致
- 设备刷写型号不匹配
- 测试用例没有覆盖主链路

## 互助记录

跨组互助建议尽量留在 Issue 评论中，便于后续追踪：

- 谁遇到了什么问题
- 谁提供了什么排查建议
- 最终根因是什么
- 如何验证已经解决
- 是否可以整理到 FAQ

## 答疑节奏建议

Issue 可以按优先级处理：

| 优先级 | 建议处理方式 |
|--------|--------------|
| P0 | 主链路阻塞，优先定位并保留排查记录 |
| P1 | 关键功能受阻，先组内排查，再请求协助 |
| P2 | 一般缺陷，适合集中答疑或按模块处理 |
| P3 | 优化建议，适合在主链路稳定后处理 |

## 经验沉淀

以下内容适合沉淀到 `07-troubleshooting/faq.md` 或小组文档中：

- 复用价值高的排查步骤
- 典型错误日志与原因
- 可复用脚本、测试集合或接口样例
- 主链路联调 checklist
- 演示环境部署注意事项
