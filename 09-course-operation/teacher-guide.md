# 教师运行建议

本文档供课程教师、企业导师和助教参考。

## 一、教师角色

| 角色 | 主要职责 |
|---|---|
| 课程负责人 | 课程目标、考核口径、节奏控制、关键决策、最终质量 |
| 企业导师 | Git/工程规范、技术架构评审、关键问题诊断、企业化点评 |
| 业务/产品导师 | 需求、用户价值、隐私边界、MVP 验收、路演表达 |
| 敏捷/过程导师 | 站会、看板、Backlog、风险暴露、团队协作证据 |
| 质量/测试导师 | 测试用例、Bug 报告、交叉测试、部署可复现性 |

教师不必每个人都成为全栈工程师。需求评审、过程监督、Issue 质量、测试证据和报告质量同样是课程质量保障。

## 二、每日节奏

| 时间 | 教师工作 |
|---|---|
| 课前15分钟 | 查看 P0/P1 Issue，确认当天重点组和风险 |
| 第一节前段 | 全班短会：当天目标、验收提醒、共性风险 |
| 上午开发时段 | 巡检看板、站会记录、主链路进度 |
| 上午中后段 | 10-15分钟微课或集中答疑 |
| 午前 | PO/SM 小会：每组最大风险、当天可交付项 |
| 下午开发时段 | 分模块诊断：后端、前端、设备、测试 |
| 下课前20分钟 | 每组更新日报、Issue 和第二天阻塞项 |

## 三、巡检重点

- 小组是否仍在推进 SOS 主链路。
- 任务是否有负责人、状态和验收标准。
- Issue 是否描述完整，P0/P1 是否被及时暴露。
- 前后端、设备、数据库是否存在等待依赖。
- 测试和文档是否从第一周开始积累。
- PO 是否维护优先级，SM 是否跟踪阻塞。
- 学生是否能说明本人贡献和验证证据。

## 四、微课触发条件

出现以下情况时，建议开 10-15 分钟短讲：

- 同一问题超过两个小组出现。
- 多数组不会写 Issue 或 Bug 报告。
- MQTT Topic/Payload 普遍不一致。
- 前后端接口字段不一致。
- 数据库状态流转设计混乱。
- P0/P1 问题影响多个小组。

## 五、答疑方式

优先使用引导式答疑：

```text
请先验证：
1. Topic 是否与后端订阅完全一致？
2. 后端日志是否出现订阅成功？
3. MQTTX 发布 Payload 是否符合接口文档？

把三项验证结果补充到 Issue 后，我们再继续判断。
```

不要直接代写学生代码。关键是训练学生定位问题、说明证据、复盘根因。
