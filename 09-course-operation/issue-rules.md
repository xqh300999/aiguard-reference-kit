# Issue 协作建议

> **用途**：本文档介绍 AiGuard 小学期项目中如何使用 Gitee Issue 进行技术求助、Bug 反馈、主链路阻塞上报和经验沉淀。Issue 使用方式可根据团队实际情况调整。

## 为什么使用 Issue

AiGuard 项目涉及后端、数据库、前端、移动端、MQTT、Home Assistant、硬件和 AI 等多条技术链路。把问题写成 Issue，有助于保留目标、现象、复现步骤、日志、排查过程和最终结论。

除账号、密钥、隐私或安全事故外，技术问题建议优先使用 Issue 记录。

## 推荐标签

仓库中建议使用中文标签。标签可按类型、优先级、模块、状态和阶段组合使用。

| 标签组 | 示例 |
|--------|------|
| 类型 | `类型：技术求助`、`类型：Bug反馈`、`类型：功能建议`、`类型：文档改进`、`类型：经验分享` |
| 优先级 | `优先级：P0主链路阻塞`、`优先级：P1关键功能`、`优先级：P2一般缺陷`、`优先级：P3优化建议` |
| 模块 | `模块：Git`、`模块：MQTT`、`模块：后端`、`模块：数据库`、`模块：前端`、`模块：鸿蒙`、`模块：硬件`、`模块：HomeAssistant`、`模块：AI`、`模块：测试` |
| 状态 | `状态：信息待补充`、`状态：学生排查中`、`状态：需要协助`、`状态：已回答`、`状态：FAQ候选` |
| 阶段 | `阶段：Sprint0`、`阶段：Sprint1`、`阶段：Sprint2`、`阶段：Sprint3`、`阶段：终期验收` |

## 优先级参考

| 等级 | 含义 | 示例 |
|------|------|------|
| P0 | 主链路无法运行 | SOS 事件无法进入后端、数据库无法连接 |
| P1 | 关键功能受阻 | 报警状态无法更新、社区端无法展示报警 |
| P2 | 功能可用但有缺陷 | 字段缺失、日志不完整 |
| P3 | 优化问题 | 样式、文案、体验优化 |

## Issue 模板

仓库已在 `.gitee/ISSUE_TEMPLATE/` 中配置表单式模板。新建 Issue 时可选择：

- 技术求助
- Bug 反馈
- 主链路阻塞
- 功能建议
- 文档改进
- 经验分享

模板会提示填写小组、模块、目标、现象、复现步骤、日志、已尝试方法、影响范围等信息。

## 关闭 Issue 的参考标准

Issue 关闭前，建议补充：

1. 根因是什么。
2. 解决办法是什么。
3. 如何验证已经解决。
4. 是否影响其他小组。
5. 是否需要整理到 FAQ。

示例：

```text
根因：后端订阅 topic 为 aiguard/v1/watch/001/alert/sos，但 MQTTX 发布到了 elder/001/event/sos。
解决：统一为 aiguard/v1/watch/001/alert/sos，并更新 docs/api.md。
验证：MQTTX 发布 SOS payload 后，后端日志出现 received sos，数据库 alerts 表新增记录，社区端报警列表可见。
关联提交：fix(mqtt): align sos topic name, Refs #12
```

## 互助与沉淀

以下内容适合沉淀为 Issue 或 FAQ：

- 能复用的环境排查方法
- MQTT Topic/Payload 对齐经验
- 前后端联调常见问题
- 硬件刷写和连接问题
- 测试用例、脚本或接口集合
- 跨组协作中的有效解决方案
