# Issue 求助与 Bug 规则

## 一、为什么统一用 Issue

本课程技术链路长，问题会很多。统一使用 Gitee Issue 可以让问题、排查过程、解决方案和评分证据沉淀下来，避免只在群聊里短暂出现。

除账号、密钥、隐私、安全事故外，技术问题优先发 Issue。

## 二、Issue 类型

| 类型 | 使用场景 |
|---|---|
| `question` | 技术求助、概念不清、环境问题 |
| `bug` | 可复现缺陷、测试发现问题 |
| `blocker` | 阻塞主链路或多人联调 |
| `docs` | 文档缺失、模板不清、FAQ 补充 |
| `feature` | 新需求或特色功能建议 |
| `review` | 请求教师或企业导师评审 |
| `showcase` | 优秀解法、脚本、可复用经验 |

## 三、优先级

| 等级 | 含义 | 示例 | 处理方式 |
|---|---|---|---|
| P0 | 主链路无法运行 | SOS 事件无法进入后端、数据库无法连接 | 当日优先处理 |
| P1 | 关键功能受阻 | 报警状态无法更新、社区端无法展示报警 | 2小时仍无法定位则请求教师介入 |
| P2 | 功能可用但有缺陷 | 字段缺失、日志不完整 | 集中答疑或组内排查 |
| P3 | 优化问题 | 样式、文案、体验优化 | 有余力再处理 |

## 四、技术求助模板

请复制 [templates/issue-question.md](templates/issue-question.md) 填写。最少必须包含：

- 小组
- 模块
- 目标
- 现象
- 复现步骤
- 截图或日志
- 已尝试方法
- 初步判断
- 影响范围

只写“老师，为什么不行”的 Issue 会被标记为 `needs-info`，补充信息后再处理。

## 五、Bug 报告模板

请复制 [templates/issue-bug.md](templates/issue-bug.md) 填写。Bug 必须可复现，并写清：

- 期望结果
- 实际结果
- 复现步骤
- 影响范围
- 测试环境
- 相关截图、日志或接口响应

## 六、教师介入规则

| 情况 | 处理方式 |
|---|---|
| 30分钟内可查资料解决 | 学生先自主探索 |
| 超过2小时仍无法定位，且影响主链路 | 教师介入指导 |
| 同一问题超过2组出现 | 教师组织10-15分钟微课 |
| P0/P1阻塞问题 | 教师及时介入 |
| 代码质量、接口规范、隐私安全问题 | 教师或企业导师进行评审 |

## 七、关闭 Issue 的标准

Issue 关闭前必须补充：

1. 根因是什么。
2. 解决办法是什么。
3. 如何验证已经解决。
4. 是否影响其他小组。
5. 是否需要加入 FAQ。

推荐关闭评论：

```text
根因：后端订阅 topic 为 aiguard/v1/watch/001/alert/sos，但 MQTTX 发布到了 elder/001/event/sos。
解决：统一为 aiguard/v1/watch/001/alert/sos，并更新 docs/api.md。
验证：MQTTX 发布 SOS payload 后，后端日志出现 received sos，数据库 alerts 表新增记录，社区端报警列表可见。
关联提交：fix(mqtt): align sos topic name, Refs #12
```

## 八、互助加分证据

以下行为可以作为增值性评价证据：

- 提交高质量 Issue
- 自主解决 P0/P1 阻塞问题
- 帮助其他小组解决问题
- 将共性问题整理为 FAQ
- 输出可复用脚本、模板或排查清单

加分必须有链接、提交、评论或文档证据。
