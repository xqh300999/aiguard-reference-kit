# 考核与过程证据说明

> **用途**：本文档说明 AiGuard 小学期项目中哪些过程证据可以帮助展示团队交付和个人贡献。具体评分口径以课程发布的考核要求为准。

## 评价维度参考

| 模块 | 评价重点 |
|------|----------|
| 考勤与职业规范 | 准时参与、沟通礼貌、设备与资料管理、职业化习惯 |
| 项目过程表现 | 站会、看板、日报、Issue、风险暴露、迭代节奏 |
| 工程质量与测试 | 仓库组织、接口文档、测试用例、Bug 报告、交叉测试、部署说明 |
| 实训报告 | 个人任务说明、技术实现、验证证据、问题解决、成长反思 |
| 终结性项目交付 | 路演表达、结果演示、项目测试、技术答辩 |
| 增值性评价 | 技术突破、高质量工程实践、团队带动、创新与作品潜力 |

## 有效证据参考

有效证据可以包括：

- Git/Gitee 提交记录
- Issue 提问、评论、关闭记录
- PR/MR 说明与评审记录
- 接口文档
- MQTT Topic/Payload 文档
- 数据库表结构与状态流转说明
- 测试用例、测试报告、Bug 报告
- 部署说明和运行截图
- Sprint 计划、站会、评审、复盘记录
- 个人实训报告中的截图、链接或演示视频时间点

以下内容通常不适合作为主要证据：

- 单纯代码行数
- 单纯提交次数
- 没有说明的截图
- 群聊口头承诺
- 无法复现的演示

## 个人贡献说明

每名成员可以从以下角度整理个人贡献：

1. 我承担了什么角色和任务。
2. 我的任务来自哪个用户故事、Issue 或 Sprint Backlog。
3. 我的上游是谁，给了我什么。
4. 我的下游是谁，我交付了什么。
5. 我如何验证自己的工作完成。
6. 证据链接或截图在哪里。

## AI 使用边界

AI 可以用于辅助学习、排错、测试用例草拟、代码审查和文档检查。但 AI 输出需要经过理解、验证和必要修改后再纳入项目。

不建议把无法解释的大段代码、编造的测试结果或含有真实 Token/密码/隐私数据的内容交给 AI 工具处理。
