## 1. API 规范

### 1.1 统一返回格式

所有 API 响应必须使用以下 JSON 格式：

```json
{
  "code": 0,
  "message": "success",
  "data": { ... },
  "ts": "2026-07-05T10:00:00Z"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| code | int | 状态码，0 成功，其他为错误 |
| message | string | 提示信息 |
| data | T | 响应数据，可为 null |
| ts | string | ISO 8601 时间戳，精确到秒 |

**成功示例：**
```json
{ "code": 0, "message": "success", "data": { "id": 1, "name": "张三" }, "ts": "2026-07-05T10:00:00Z" }
```

**错误示例：**
```json
{ "code": 401, "message": "用户名或密码错误", "data": null, "ts": "2026-07-05T10:00:00Z" }
```

### 1.2 HTTP 方法语义

| 方法 | 语义 | 示例 |
|------|------|------|
| GET | 查询资源（列表/详情） | `GET /api/elderly`、`GET /api/elderly/1` |
| POST | 创建资源 | `POST /api/v1/elderly` |
| PUT | 全量更新资源 | `PUT /api/v1/elderly` |
| DELETE | 删除资源 | `DELETE /api/v1/elderly/1` |
| PATCH | 部分更新资源 | `PATCH /api/v1/elderly/1/status` |

### 1.3 URL 命名规范

- **一律使用小写英文 + 连字符**
- **名词复数形式**表示资源集合
- **嵌套资源用 `/` 层级表示**
- **所有接口以 `/api/v1/` 开头**

```
/api/v1/auth/login                  ← 认证
/api/v1/communities                 ← 社区列表
/api/v1/communities/1               ← 单个社区
/api/v1/elderly                     ← 老人列表
/api/v1/elderly/1/alerts            ← 老人的告警记录（嵌套）
/api/v1/elderly/1/alerts/2          ← 老人的某条告警详情
```

### 1.4 状态码定义

| code | 含义 | 说明 |
|------|------|------|
| 0 | 成功 | 请求正常处理 |
| 400 | 参数错误 | 请求参数校验失败 |
| 401 | 未认证 | Token 缺失或无效 |
| 403 | 无权限 | 角色无权访问该接口 |
| 404 | 资源不存在 | 查询的资源未找到 |
| 500 | 服务端错误 | 服务器内部异常 |

### 1.5 认证方式

所有需要登录的接口（除 `/api/v1/auth/login` 外）必须在请求头携带 JWT Token：

```
Authorization: Bearer <token>
```

### 1.6 分页规范（后续扩展）

```
GET /api/v1/elderly?page=1&size=20

响应：
{
  "code": 0,
  "message": "success",
  "data": {
    "records": [...],
    "total": 100,
    "page": 1,
    "size": 20
  },
  "ts": "2026-07-05T10:00:00Z"
}

### 1.7 接口版本升级策略

- 当前版本为 `/api/v1/`
- **兼容性变更**（新增字段、新增接口）在 v1 内直接加，不升级版本号
- **破坏性变更**（字段改名、字段类型变化、删除接口）必须开 `/api/v2/`
- v2 发布后，v1 至少保留一个 Sprint 周期供前端迁移

## 2. 数据库规范

### 2.1 命名规范

| 项目 | 规范 | 示例 |
|------|------|------|
| 数据库名 | 小写+下划线 | `aiguard` |
| 表名 | 小写+下划线，单数名词 | `community`、`user`、`elderly` |
| 字段名 | 小写+下划线 | `real_name`、`community_id` |
| 主键 | `id` BIGSERIAL | `id BIGSERIAL PRIMARY KEY` |
| 外键 | 引用表名+`_id` | `community_id`、`elderly_id` |
| 时间字段 | `_at` 后缀 | `created_at`、`updated_at`、`resolved_at` |

### 2.2 建表规范

- **数据库使用 openGauss**（开发环境可使用 PostgreSQL 兼容）
- **所有表必须有 `id` 和 `created_at` 字段**
- 枚举字段使用 VARCHAR + CHECK 约束 或 Java 枚举映射
- 大文本字段使用 `TEXT` 类型
- 外键必须建立索引

```sql
CREATE TABLE example (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    status VARCHAR(20) DEFAULT 'ACTIVE',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 2.3 时间字段处理

- Java 实体类中使用 `LocalDateTime` 类型
- 数据库时间类型使用 `TIMESTAMP`（PG 统一时间类型）
- MyBatis-Plus 自动映射下划线→驼峰（`created_at` ↔ `createdAt`）
- 前后端传输使用 `ISO 8601` 格式（如 `2026-07-05T10:00:00Z`）

### 2.4 密码存储

- **用户密码必须使用 BCrypt 加密**
- 禁止明文存储密码
- Java 后端用 `BCryptPasswordEncoder` 处理

### 2.5 环境变量与敏感信息管理

- `.env`、`application-local.yml`、`*.secret.*` 等含敏感信息的文件**禁止提交到 Git**，已加入 `.gitignore`
- 数据库密码、JWT 密钥、MQTT 用户名密码等通过环境变量或 `application.yml` 占位符 `${...}` 注入
- 每个开发者在本地创建 `application-local.yml`（不提交），在其中覆盖开发环境配置

```yaml
# application-local.yml（不提交到Git）
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/aiguard
    username: dev
    password: dev123
jwt:
  secret: my-local-dev-secret-key
mqtt:
  username: dev
  password: dev123
```

## 3. 后端开发规范

### 3.1 包结构

```
com.elderlycare
├── config/           ← 配置类（SecurityConfig、MqttConfig）
├── controller/       ← REST 控制器
├── service/          ← 业务逻辑层
├── mapper/           ← MyBatis-Plus Mapper 接口
├── model/
│   ├── entity/       ← 数据库实体类
│   ├── dto/          ← 请求/响应 DTO
│   └── enums/        ← 枚举类
├── security/         ← JWT 认证相关
└── common/           ← 公共类（Result、GlobalExceptionHandler）
```

### 3.2 三层分层原则

```
Controller → Service → Mapper
    ↑           ↑         ↑
  接收请求     业务逻辑   数据库操作
  参数校验     事务管理    SQL 映射
  返回结果     调用 Mapper
```

- **Controller 层**：不做业务逻辑，只做参数校验 + 调用 service + 返回 Result
- **Service 层**：只做业务逻辑，不直接处理 HTTP 请求
- **Mapper 层**：只做数据库操作，不包含业务逻辑

### 3.3 Controller 规范

```java
@RestController
@RequestMapping("/api/v1/资源名")
@RequiredArgsConstructor
public class XxxController {
    private final XxxService xxxService;

    @GetMapping("/{id}")
    public Result<Xxx> getById(@PathVariable Long id) {
        return Result.success(xxxService.getById(id));
    }

    @GetMapping
    public Result<List<Xxx>> list(...) {
        return Result.success(xxxService.listAll());
    }

    @PostMapping
    public Result<Xxx> create(@RequestBody @Valid Xxx xxx) {
        return Result.success(xxxService.create(xxx));
    }

    @PutMapping
    public Result<Void> update(@RequestBody Xxx xxx) {
        xxxService.update(xxx);
        return Result.success();
    }

    @DeleteMapping("/{id}")
    public Result<Void> delete(@PathVariable Long id) {
        xxxService.delete(id);
        return Result.success();
    }
}
```

### 3.4 异常处理

- **禁止在 Controller 中 try-catch 后返回 null**
- 业务异常直接在 Service 层抛出 `RuntimeException`
- `GlobalExceptionHandler` 统一捕获并返回 `Result.error()`
- 错误信息要清晰可读（中文），不要抛技术栈信息给前端

```java
// 正确做法
if (user == null) {
    throw new RuntimeException("用户名或密码错误");
}

// 错误做法
if (user == null) {
    return Result.error("error");
    // 或者 return null;
}
```

### 3.5 日志规范

- 使用 SLF4J 日志门面
- 异常必须打印完整堆栈
- 关键操作（创建/更新/删除）记录操作人和操作内容

```java
private static final Logger log = LoggerFactory.getLogger(XxxService.class);
log.error("操作失败", e);
log.info("用户 {} 创建了告警 {}", userId, alertId);
```

### 3.6 MyBatis-Plus 使用规范

- **所有 Mapper 继承 `BaseMapper<Entity>`**
- **所有实体类使用 `@TableName` 注解**
- **主键使用 `@TableId(type = IdType.AUTO)`**
- 使用 `LambdaQueryWrapper` 构建查询条件（避免硬编码字段名）

```java
// 正确
new LambdaQueryWrapper<User>()
    .eq(User::getCommunityId, communityId);

// 错误
new QueryWrapper<User>()
    .eq("community_id", communityId);
```

## 4. 前端开发规范

### 4.1 技术选型确认

| 项 | 选型 |
|----|------|
| 框架 | Vue 3 |
| 构建工具 | Vite |
| UI 组件库 | Element Plus |
| 状态管理 | Pinia |
| HTTP 请求 | Axios |
| 图表 | ECharts（可选） |

### 4.2 API 请求封装

所有 API 请求必须通过 `utils/request.js` 发送，该模块已统一处理：

- **自动携带 Token**：从 `localStorage` 读取并注入 `Authorization` 头
- **统一错误处理**：401 自动跳转登录页，其他错误显示 ElMessage
- **自动解包**：`request.get()` 返回的是 `res.data` 而非 `res`

```javascript
// 正确用法
import request from '../../utils/request'

const fetchList = async () => {
  const res = await request.get('/communities')
  list.value = res.data  // ✅ 直接拿到 data
}

// 错误用法
const fetchList = async () => {
  // ❌ 不应手动拼接 URL
  // ❌ 不应手动处理 Token
  // ❌ 不应自己做错误弹窗
}
```

### 4.3 组件命名规范

| 类型 | 命名 | 示例 |
|------|------|------|
| 页面文件 | 大驼峰 + `View` 后缀 | `LoginView.vue`、`ElderlyMgt.vue` |
| 公共组件 | 大驼峰，功能前缀 | `ElderlyTable.vue`、`AlertCard.vue` |
| 布局组件 | `XxxLayout.vue` | `AdminLayout.vue`、`WorkerLayout.vue` |

### 4.4 vue-router 路由规范

- **路径全小写，多词用连字符**
- 管理后台路由前缀 `/admin`，社区人员 `/worker`，家人 `/family`
- 路由守卫统一处理登录校验和角色判断

```javascript
{
  path: '/admin',
  component: AdminLayout,
  meta: { role: 'ADMIN' },
  children: [
    { path: 'elderly', component: ElderlyMgt }
  ]
}
```

### 4.5 Pinia 状态管理

- 用户认证信息统一使用 `store/auth.js`
- 禁止在组件中直接读写 `localStorage`
- 通过 `useAuthStore()` 获取/修改认证状态

```javascript
import { useAuthStore } from '@/store/auth'
const auth = useAuthStore()
console.log(auth.realName)   // ✅
auth.logout()                // ✅ 自动清除 localStorage
```

### 4.6 代码风格

- 使用 Vue 3 组合式 API（`<script setup>`）
- 禁止使用 Options API（`data()`、`methods`）
- 组件内按顺序组织：`template` → `script setup` → `style scoped`

### 4.7 代码格式化

- 使用 ESLint 进行代码检查，Prettier 进行代码格式化
- `package.json` 中配置以下脚本：

```json
{
  "scripts": {
    "lint": "eslint . --fix",
    "format": "prettier --write src/"
  }
}
```

- 提交前运行 `npm run lint`，确保无错误
- VS Code 推荐安装 ESLint + Prettier 插件，开启保存时自动格式化

## 5. Git 规范

### 5.1 分支策略

```
main          ← 稳定版本，仅从 dev 合并
  └── dev     ← 开发主分支，所有功能分支合并到此
       ├── feat/xxx     ← 功能分支（feat/login、feat/alert）
       ├── fix/xxx      ← 修复分支（fix/login-bug）
       └── refactor/xxx ← 重构分支
```

### 5.2 Commit Message 规范

```
type(scope): description
```

| type | 适用场景 |
|------|---------|
| feat | 新功能 |
| fix | Bug 修复 |
| refactor | 重构 |
| docs | 文档 |
| style | 格式（不影响功能） |
| test | 测试 |
| chore | 构建/工具 |

示例：
```
feat(auth): add login API
fix(alert): correct NPE in AlertController
docs(api): update alert response format
```

### 5.3 基本操作流程

```bash
# 1. 从 dev 创建功能分支
git checkout dev
git pull
git checkout -b feat/my-feature

# 2. 开发并提交
git add <文件>
git commit -m "feat(scope): add xxx"

# 3. 合并回 dev
git checkout dev
git pull
git merge feat/my-feature

# 4. 推送
git push origin dev
```

- **禁止直接向 main 分支提交代码**
- **每天开始工作前先 pull 最新 dev**
- **功能分支完成后必须合并回 dev**

## 6. MQTT 协议规范

### 6.1 Broker

- **使用 Mosquitto**（非 EMQX）
- 默认端口：1883（TLS 端口 8883）
- SOS 消息必须使用 **QoS 2**（恰好一次）

### 6.2 Topic 命名

```
aiguard/v1/{设备类型}/{设备ID}/{领域}/{动作}
```

| 变量 | 说明 | 示例值 |
|------|------|--------|
| `设备类型` | 设备分类 | `watch`(手表)、`panel`(大屏)、`gateway`(网关) |
| `设备ID` | 设备唯一标识 | 手表 IMEI 或 MAC 地址 |
| `领域` | 业务领域 | `alert`、`health`、`system` |
| `动作` | 操作类型 | `sos`、`heartbeat`、`sensor`、`command` |

| Topic | 方向 | 说明 | QoS |
|-------|------|------|-----|
| `aiguard/v1/watch/{id}/alert/sos` | 设备→服务器 | SOS 告警（紧急最高） | **2** |
| `aiguard/v1/watch/{id}/alert/fall` | 设备→服务器 | 跌倒检测告警 | 1 |
| `aiguard/v1/watch/{id}/alert/low_battery` | 设备→服务器 | 低电量告警 | 1 |
| `aiguard/v1/watch/{id}/health/heartbeat` | 设备→服务器 | 设备心跳 | 1 |
| `aiguard/v1/watch/{id}/health/sensor` | 设备→服务器 | 传感器数据上报 | 1 |
| `aiguard/v1/panel/{id}/display/command` | 服务器→设备 | 大屏显示指令 | 1 |
| `aiguard/v1/gateway/{id}/system/ota` | 服务器→设备 | OTA 升级指令 | 0 |

### 6.3 消息格式（JSON）

**SOS 告警上报（watch → 服务器，QoS 2）：**
```json
{
  "type": "sos",
  "deviceId": "watch_a1b2c3d4",
  "timestamp": "2026-07-05T10:30:00Z",
  "data": {
    "elderlyId": 1,
    "lat": 30.5728,
    "lng": 104.0668
  }
}
```

**心跳上报（设备→服务器）：**
```json
{
  "type": "heartbeat",
  "deviceId": "watch_a1b2c3d4",
  "timestamp": "2026-07-05T10:30:00Z",
  "data": {
    "battery": 85,
    "rssi": -65
  }
}
```

**跌倒检测上报（watch → 服务器）：**
```json
{
  "type": "fall",
  "deviceId": "watch_a1b2c3d4",
  "timestamp": "2026-07-05T10:30:00Z",
  "data": {
    "elderlyId": 1,
    "impact": 9.8,
    "angle": 75.3
  }
}
```

**指令下发（服务器→设备）：**
```json
{
  "type": "command",
  "deviceId": "watch_a1b2c3d4",
  "timestamp": "2026-07-05T10:30:00Z",
  "data": {
    "action": "vibrate",
    "params": { "duration": 3000 }
  }
}
```

## 8. 协作规范

### 8.1 平台与仓库

- **代码托管：Gitee**（非 GitHub）
- **仓库结构：** 统一仓库，按模块分目录

### 8.2 分支策略

```
main            ← 生产就绪（每个 Sprint 结束时合并）
  └── dev       ← 开发主线（日常集成）
       ├── feat/xxx  ← 功能分支（从 dev 切出）
       ├── fix/xxx   ← 修复分支
       └── docs/xxx  ← 文档分支
```

- 禁止直接 push 到 `main` 和 `dev`
- 功能开发从 `dev` 开分支，完成后提 PR/MR
- 每个 PR 必须至少一人 Review 通过后合并

**Commit 格式统一见第 5.2 节。** 此处不再重复。

### 8.4 接口文档同步

- **后端开发 API 后，必须在 Gitee Wiki / 共享文档更新接口文档**
- 文档内容包括：URL、请求方法、请求参数、响应格式
- 前端开发前先确认接口文档已就绪

### 8.5 前后端联调约定

- **前后端并行开发期间，前端使用 Mock 数据**
- 后端接口就绪后，前端切换为真实接口
- 联调发现的问题统一记录在共享文档

### 8.6 前后端数据格式对应

| 数据库 | Java | 前端 |
|--------|------|------|
| `created_at` | `createdAt` | `createdAt` |
| `VARCHAR('SOS')` | `AlertTypeEnum` | 字符串 `"SOS"` |
| `TIMESTAMP` | `LocalDateTime` | `"2026-07-05T10:00:00Z"` |

### 8.7 每日站会

- 每天固定时间，15 分钟
- 每人回答：昨天做了什么 / 今天做什么 / 有什么阻塞
- **不展开讨论问题**，站会后单独沟通

### 8.8 代码评审（Code Review）

- 合并到 dev 前必须至少让另一人 Review

**Review Checklist（逐项检查）：**

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | 符合编码规范 | 命名风格、包结构、缩进、空行等是否与团队规范一致 |
| 2 | 无明显 Bug | 空指针、数组越界、未关闭资源、并发问题 |
| 3 | 边界处理 | 空数据、超长输入、null 值是否处理 |
| 4 | 安全性 | SQL 注入（MyBatis 用 `#{}` 非 `${}`）、XSS、敏感信息不写日志 |
| 5 | 性能 | 避免 N+1 查询，列表接口加索引，批量操作用批量接口 |
| 6 | 过度设计 | 代码是否过度抽象、不必要的设计模式 |
| 7 | 重复代码 | 是否有可抽取的公共方法或组件 |
| 8 | 日志 | 关键操作（创建/更新/删除/异常）有日志 |
