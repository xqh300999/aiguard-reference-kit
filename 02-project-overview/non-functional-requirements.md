# 非功能需求

> **用途**：本文档定义 AiGuard 系统在性能、安全、可靠性、可扩展性、可用性等方面的质量要求。这些要求是系统应该达成的目标，具体实现方式不限。

---

## 一、性能要求

### 1.1 响应时间
- **API 响应时间**：普通 REST API 请求 P95 响应时间 ≤ 500ms
- **实时数据推送**：设备数据上报到前端展示延迟 ≤ 2 秒（正常网络条件下）
- **告警触发延迟**：从事件发生到告警通知发出 ≤ 3 秒
- **页面加载时间**：Web 仪表盘首屏加载 ≤ 3 秒
- **设备控制响应**：从 App/面板发出指令到设备执行 ≤ 2 秒

### 1.2 吞吐量
- 支持至少 50 台设备同时在线（单家庭/小规模部署）
- 支持每秒至少 100 条消息处理
- 支持至少 10 个用户同时访问系统
- 对于养老机构等大规模场景，可横向扩展支持更多设备和用户

### 1.3 数据处理
- 时序数据支持至少 30 天在线查询
- 历史数据可归档存储（支持更长时间的回溯，实现方式不限）
- 支持并发数据写入，不丢失数据
- AI 日报生成时间 ≤ 30 秒/份

> **说明**：以上性能指标为参考目标。对于 MVP 阶段，可适当降低要求；对于家庭场景，实际并发量可能远低于上述数值。

---

## 二、可靠性要求

### 2.1 可用性
- 系统整体可用性目标：99% 以上（允许少量计划内维护停机）
- 核心告警功能必须可靠，不能漏报
- 允许非核心功能（如 AI 对话、历史查询）短暂不可用，但不能影响核心安全功能

### 2.2 容错性
- 单个设备故障不影响系统整体运行
- 数据库/中间件故障要有优雅降级机制
- 网络中断恢复后，设备能自动重连并续传关键数据
- 后端服务重启后不丢失关键数据

### 2.3 数据可靠性
- 用户数据、告警记录不能丢失
- 关键操作有日志记录，可追溯
- 定期数据备份（实现方式不限：自动备份脚本、数据库备份等）

---

## 三、安全要求

### 3.1 认证与授权
- 所有 API 访问需要认证
- 设备接入需要设备级认证（Token/证书/用户名密码等，实现方式不限）
- 不同用户角色有不同权限（RBAC 或其他权限模型）
- 会话管理：Token 过期机制、登出机制
- 密码不能明文存储，需要加密/哈希

### 3.2 数据安全
- 数据传输加密：HTTPS/WSS/MQTT over TLS
- 敏感数据（密码、Token 等）存储加密
- 摄像头视频流等隐私数据优先本地处理，传输和存储要有保护
- 防止 SQL 注入、XSS、CSRF 等常见 Web 攻击
- 接口访问频率限制，防止暴力破解

### 3.3 隐私保护
- **隐私保护是养老看护系统的重中之重**
- 数据最小化原则：只采集必要的数据
- 用户知情权：明确告知采集什么数据、用于什么用途
- 用户控制权：用户可以查看、导出、删除自己的数据
- 隐私访问审计：谁访问了敏感数据有日志记录
- 摄像头等视频数据：优先本地存储和处理，用户可配置是否上云

### 3.4 告警安全
- 紧急告警（SOS、跌倒等）不能漏报
- 告警通知要有确认机制
- 防止误报过多导致「狼来了」效应（合理的防抖和确认机制，实现方式不限）

---

## 四、可扩展性要求

### 4.1 设备扩展
- 支持方便地添加新类型的设备
- 设备驱动/协议适配层可插拔（实现方式不限）
- 不需要改核心代码就能接入新设备类型

### 4.2 功能扩展
- 规则引擎支持自定义规则扩展
- 支持添加新的 AI 能力
- 支持对接新的通知渠道（除了 App 推送，还可以是电话、短信、微信等）

### 4.3 部署扩展
- 支持单设备/单家庭小规模部署（如家用服务器、树莓派等）
- 支持云部署，可横向扩展支持多家庭（养老机构等场景）
- 容器化部署，环境一致性好

---

## 五、可用性与易用性要求

### 5.1 用户界面易用性
- **老人使用的设备（面板、手表）要简单**：大字体、高对比度、操作逻辑直接
- 紧急按钮（SOS）要醒目、容易触发，防止误触但关键时刻能按到
- 告警提示要醒目：声音洪亮、屏幕闪烁、震动（手表）
- 家属端界面信息清晰，重点突出告警和关键状态
- 日常操作步骤少，不超过 3 步完成常用操作

### 5.2 部署与运维易用性
- 提供一键启动/部署脚本（Docker Compose 或其他方式）
- 有清晰的部署文档和运行手册
- 配置简单，提供合理的默认配置
- 日志清晰，问题容易排查
- 支持设备快速配网（WiFi 配网、蓝牙配网等，实现方式不限）

### 5.3 可观测性
- 关键操作有日志记录
- 系统运行状态可监控（服务健康、设备在线率等）
- 错误有明确的错误信息，便于定位问题
- 有运行状态仪表盘（可选）

---

## 六、兼容性要求

### 6.1 设备兼容性
- 支持主流浏览器访问 Web 仪表盘（Chrome、Firefox、Safari、Edge 最新版本）
- 中控面板固件适配 ESP32-P4 硬件
- 手表固件适配 ESP32-S3 硬件
- 支持与 HomeAssistant 兼容对接（HA 版本兼容性，实现方式不限）

### 6.2 网络兼容性
- 支持家庭 WiFi 网络
- 设备支持网络不稳定场景（弱网、断网重连）
- MQTT 协议为主，可扩展其他协议

---

## 七、可维护性要求

### 7.1 代码质量
- 代码结构清晰，分层合理
- 关键逻辑有注释
- 有单元测试（核心业务逻辑测试覆盖率建议 60%+，实现方式不限）
- 有基本的集成测试/契约测试

### 7.2 文档
- 有部署文档
- 有 API 文档（实现方式不限：Swagger/OpenAPI、手写文档等）
- 有配置说明文档
- 固件有编译、烧录说明

### 7.3 开发效率
- 本地开发环境容易搭建
- 提供模拟设备脚本，不需要真实硬件也能开发调试
- 代码改动后能快速看到效果（热重载等，根据技术栈实现）

---

## 八、国际化与本地化（可选）

- 界面默认中文
- 预留国际化扩展能力（可选，不是 MVP 必须）
- 时间、日期、数值格式符合中文习惯

## 质量优先级参考

在资源有限的情况下，非功能需求也有优先级：

| 优先级 | 要求 | 原因 |
|--------|------|------|
| P0 必须 | 告警可靠性、数据安全、隐私保护、认证授权 | 安全类问题是底线，不能出问题 |
| P0 必须 | 核心功能响应速度、设备稳定性 | 用户体验基础 |
| P1 应该 | 可扩展性、可维护性、可观测性 | 长期开发效率 |
| P1 应该 | 易用性（老人端） | 老人用不了就是废品 |
| P2 可以 | 高吞吐量支持、国际化、监控仪表盘 | MVP 阶段可以简化 |

---

## 实现提示

非功能需求不是靠某一个模块实现的，而是渗透在整个系统的设计和开发中：

- **性能**：合理的数据库索引、缓存策略、异步处理
- **可靠性**：重试机制、降级策略、数据备份
- **安全**：框架提供的安全机制、代码审计、加密
- **可扩展性**：分层架构、接口抽象、插件化设计
- **易用性**：站在用户角度思考，UI/UX 设计

> **提示**：以上非功能需求是质量目标，不是硬性约束。实现方式不限。MVP 阶段可以优先保证核心安全和可靠性，其他非功能需求可以在后续迭代中逐步完善。
